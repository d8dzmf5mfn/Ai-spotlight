# Workflow

This project is developed with significant LLM (AI coding assistant) involvement.
LLM sessions can be **interrupted or wiped at any time** — context loss is a real
risk, not a theoretical one. These rules exist because git is the only durable
memory the project has.

## The one rule

> **Commit early, commit often. Never end a session with uncommitted work.**

If a session is interrupted mid-edit, everything in the working tree that
hasn't been committed is gone. The next session can recover git history,
but cannot recover files that were never saved.

## At the end of every AI-assisted session

Run:

```bash
./scripts/snapshot.sh
```

This shows staged, unstaged, and untracked files **without committing** —
just a status report. Read it. For every line that says "WARNING" or shows
untracked files, make an explicit decision: commit it, gitignore it, or
delete it.

If you want to stage everything first and then review:

```bash
./scripts/snapshot.sh --add
```

This stages all changes but still does not commit. After reviewing the
output, commit manually:

```bash
git commit -m "..."
```

## What belongs in git

| Type | In git? | Why |
|---|---|---|
| Source code (`.swift`) | ✅ | The whole point of the repo |
| Build scripts (`scripts/*.sh`) | ✅ | Reproducible builds |
| `README.md`, `docs/`, `LICENSE` | ✅ | Project description |
| `Package.swift`, `.gitignore` | ✅ | Repo metadata |
| `build/AI Spotlight.app` | ❌ | Regenerated by `make_app.sh` |
| `build/*.dmg` | ❌ | Regenerated by `make_dmg.sh`, not notarized |
| `.build/` (SPM artifacts) | ❌ | Regenerated by `swift build` |
| `Secrets.plist`, API keys | ❌ | Use Keychain instead |
| `~/Documents/AI-Spotlight-Testing.md` | ❌ | Local notes, not project state |

## What about WIP code?

If you have half-finished work that shouldn't be committed yet, **branch**:

```bash
git checkout -b wip/<short-name>
# do the work
git add -A && git commit -m "wip: <what's done>"
```

The branch is durable. The session can be wiped. When you come back,
`git checkout wip/<short-name>` and pick up where you left off.

## LLM context recovery

If a session is lost and you start a new one, point the LLM at:

1. `git log --oneline -20` — recent history
2. `git status` — current working state
3. `docs/WORKFLOW.md` (this file) — project conventions
4. `.hermes/` (if present) — LLM-side notes managed by the AI tool

Do **not** rely on the LLM remembering anything from the previous session.
It doesn't.

## Why no `.hermes/` in git?

This project's `.hermes/` directory is a **per-machine, per-user** LLM
tooling folder (skills, notes, debug logs). It is intentionally not
committed. See `.gitignore` if you want to verify.
