## 3. Runtime execution model

This section describes the actual code paths a user query takes, verified by reading source.

### 3.1 Query pipeline (keystroke → results)

```
User types in SpotlightPanel (NSTextField)
  │
  ▼
SearchField → AppState.onQueryChange(raw)
  │
  ├─ Cancel previous searchTask
  ├─ Start 0.6s debounce timer
  │
  ▼ (after 600ms idle)
AppState.runSearch(query)
  │
  ├─ CommandMatcher.match(query) — fast path for "settings", etc.
  │   └─ If match: results = [command]; return
  │
  ├─ QueryInterpreter.interpret(query)              [Verified: Sources/AISpotlightKit/QueryInterpreter.swift:41]
  │   ├─ Cache hit (LRU 100) → return cached Intent
  │   ├─ Tier 2: LLMIntentRouter (if llmRouter != nil && aiProvider != nil)
  │   │   └─ Currently: SKIPPED (see §4.2)
  │   └─ Tier 3: QueryParser.parse(raw) → .findFile | .openApp | .unknown
  │       └─ If .unknown AND aiProvider != nil → .ask(query, contextURLs: [])
  │
  ├─ SearchOrchestrator.run(intent)                [Verified: Sources/AISpotlightKit/SearchOrchestrator.swift:11]
  │   ├─ Fan out intent to all providers in parallel via TaskGroup
  │   │   ├─ FileSystemProvider (MDQuery for filename/path/date/kind)
  │   │   ├─ AppProvider (ls /Applications)
  │   │   └─ ContentSearchProvider (MDQuery for kMDItemTextContent)
  │   └─ ResultMerger.merge(buckets) → ranked [SearchResult]
  │
  ├─ If intent is .ask → save top-5 file URLs as `lastSearchContextURLs`
  │
  └─ Update @Published `results` → SwiftUI SearchWindowView re-renders
```

### 3.2 LLM pipeline (Enter key → reply)

```
User presses Enter with .ask intent (or empty text with provider configured)
  │
  ▼
AppState.runLLMAsk(query, contextURLs)
  │
  ├─ Increment llmGeneration (monotonic counter; older tasks get discarded)
  │
  ├─ If useTools == true (default true, user-configurable in Settings)
  │   └─ LLMConversationService.askWithTools(query, history, context, toolRegistry, maxToolTurns: 2)
  │       ├─ System prompt asks LLM to call ≤1 tool per question (Phase 5-J fix)
  │       ├─ LLM may call any of 7 registered tools (search_files, open_file,
  │       │   list_apps, run_shell, read_file, clipboard_get, clipboard_set)
  │       ├─ 3 of those require user consent (run_shell, read_file, clipboard_set)
  │       │   └─ AppState.requestUserConsent → pendingConsent @Published → modal dialog
  │       └─ Returns LLMResult { finalAnswer, toolCalls }
  │
  └─ Else (useTools == false) → LLMConversationService.askStreamingWithHistory
      └─ AsyncThrowingStream of text chunks → accumulate into llmReply
```

## 4. Architectural realities vs. README claims

### 4.1 README claim: "Three-tier routing (cache → LLM → rules)"

- **Reality:** Two tiers. Cache + rules. LLM tier exists in code (`LLMIntentRouter` is a real type with tests) but is **not wired** in `main.swift:124-125`.
- The 3-tier architecture described in the README **is the original 4.2 design**. It was **deliberately disabled in Phase 4.2.5** for stability (every keystroke was firing a separate LLM call, causing latency spikes and Ollama CPU pressure). The fix was to move LLM invocation to a single call on Enter via `runLLMAsk`, not on every keystroke via the intent router.
- **This is a real divergence.** README is not lying — it describes an earlier design — but it does not flag the change.

### 4.2 `LLMIntentRouter` state

- ✅ VERIFIED — `Sources/AISpotlight/main.swift:124-125`:
  ```swift
  // let llmRouter: LLMIntentRouter? = ai.map { LLMIntentRouter(provider: $0) }
  let llmRouter: LLMIntentRouter? = nil
  ```
- ✅ VERIFIED — `QueryInterpreter` accepts `llmRouter: LLMIntentRouter?` as a constructor parameter; the parameter is genuinely honored in `interpret()` (line 58: `if let router = llmRouter, let _ = aiProvider { ... }`).
- ✅ VERIFIED — `Tests/AISpotlightKitTests/LLMIntentRouterTests.swift` exists with 10 tests, all passing. The router is **not broken**; it is **unwired**.
- **Interpretation:** This is **intentional disablement via architectural removal at the wiring point**, not a regression. The 4.2.5 commit message in `main.swift:112-123` explains the rationale in detail.

## 8. Stability verdict

### CONDITIONAL STABLE — proceed with caveats

**What this audit can confirm:**
- The Swift module graph compiles cleanly (release build in 6.3s)
- 171/171 unit tests pass with 0 failures
- The runtime execution model is well-defined and follows the code paths in §3
- The `LLMIntentRouter`-disabled state is **intentional, documented, and reversible** (one-line `main.swift:124` change re-enables it)

**What this audit CANNOT confirm:**
- User-facing search quality (latency, ranking, recall) — requires GUI runtime testing
- LLM intent classification correctness — untested at runtime because the router is disabled
- End-to-end `AppState` behavior — 853-line orchestrator with 0 dedicated tests
- Performance under realistic load (RSS, debounce jitter, large file corpora)
- Behavior on macOS versions other than the build host (macOS 27.0 SDK on macOS 14.0 deployment target)
- Provider-specific behavior (Ollama / DeepSeek / OpenAI / etc.) under failure modes

**Conditions for proceeding safely with further engineering work:**
1. The README drift (§4.1, §4.3, §4.4) should be resolved before any **external** announcement or release — internal engineers can read code, external users cannot
2. The dead-code `LLMIntentRouter` (§4.2) is acceptable **only as long as** the intentionality is documented. If a new contributor arrives, they should not "fix" the `// let llmRouter = ...` line without understanding the 4.2.5 trade-off
3. Any new feature (e.g. Step-2 SQLite sync) should be tested with a `git blame`-aware eye: the project's history includes a 5GB-RSS regression (3.1.5), a per-keystroke LLM explosion (4.2.5), and a tool-calling loop (5-J). These are the failure modes most likely to recur

## 11. Post-audit risk additions (2026-06-17, post-review)

Two structural risks were identified during user review of the original audit (§1–§10). Both were re-verified by direct code read before being added here.

### 11.1 `AppState` is a cohesive wide class, not a god object (severity: MEDIUM) — revised 2026-06-17 after verification

**Revision:** the original §11.1 framed this as a god object (HIGH severity, god-object anti-pattern). After reading the relevant sections (init §168-225, runLLMAsk §331-360, activate §612-743, classifyLLMError §494-550) the picture is more nuanced. `AppState` is wide (853 lines, 10 distinct responsibilities) but the responsibilities are **layered** rather than entangled. The "god object" framing overstates the coupling risk.

- ✅ VERIFIED — `Sources/AISpotlight/App/AppState.swift` is 853 lines, single `@MainActor` `ObservableObject` class. 10 distinct public/private responsibilities:

  1. Search orchestration (`runSearch`, 225–284)
  2. LLM orchestration with two parallel paths (`runLLMAsk` with tools / without tools, 332–468)
  3. LLM error classification (`classifyLLMError`, 494–550) — including domain-specific error-code translation
  4. NSWorkspace observer for Ollama lifecycle (`handleOllamaAppTerminated`, 183–194)
  5. **Global mutable state** via `static var lastOllamaQuitTimestamp` (564) — used to disambiguate user-quit vs idle-exit
  6. Consent gate (`requestUserConsent` / `respondToConsent`, 792–825)
  7. Activation logic with 3 priority arms (`activate`, 612–743) — including debounce flush, command dispatch, file-open side effects
  8. LLM-reply path extraction regex (`extractPaths`, 759–778) — file-existence filtering
  9. UI state binding (`moveSelection`, `closePanel`, `openLLMReplyPath`, 585–785)
  10. Ollama-not-running 错误分类 — branches inside `classifyLLMError`

- ✅ VERIFIED — `AppState` has 0 dedicated unit tests. The only coverage of its code paths is indirect (via `QueryInterpreter`, `LLMConversationService` etc. tests that do not exercise `AppState`'s orchestration).
- ✅ VERIFIED — `AppState` has `static var lastOllamaQuitTimestamp` (a global mutable), and the source comment explicitly acknowledges the singleton assumption: *"AI Spotlight has exactly one AppState (singleton pattern via main.swift), so it's fine in practice."* This is a load-bearing assumption not enforced anywhere.

**Why this matters for engineering work:**

Any new feature (Step-2 SQLite sync, Step-3 ranking signals, semantic cache) will face pressure to be wired into `AppState` because that's where the orchestrator lives. The codebase already has 3 historical failure modes caused by adding responsibilities to a single type without bound (5GB-RSS 3.1.5, per-keystroke LLM 4.2.5, tool-calling loop 5-J). Adding more weight to `AppState` increases the risk of a fourth.

**Layering evidence (verified 2026-06-17):**

- **3 orchestration methods** (`runSearch`, `runLLMAsk`, `activate`) are the **sole writers** of all `@Published` state. Each `@Published` field has exactly one owner.
- **Static mutable state** is limited to `lastOllamaQuitTimestamp` (line 564) — written by `handleOllamaAppTerminated`, read by `classifyLLMError`. Two call sites total.
- **Cross-responsibility coupling is narrow and unidirectional**: side-effect handler → orchestration reads timestamp, orchestration writes `@Published`. No reverse coupling.
- **`@MainActor`** enforces compile-time serialization of all state access. Cascading mutations are statically impossible.
- **Async cancellation** (`llmGeneration` counter + `Task?.cancel()`) is centralized in `runLLMAsk`; not scattered.

**This is more like a "cohesive wide class" than a "god object".** The distinction matters:
- God object: responsibilities are tangled, state mutations cascade unpredictably, changing one field ripples.
- Cohesive wide class: many responsibilities but layered (state / orchestration / UI binding), each with a clear single owner, cross-layer coupling is narrow.

`AppState` matches the second pattern.

**Recommended action (revised; was: extract coordinator):**
- Adding **features** to `AppState` (new orchestration logic, new state fields) is acceptable with caution: each new `@Published` field should have one owner, and any new cross-responsibility state should be reviewed for the narrow-coupling pattern above.
- A **full extraction refactor** (split into SearchOrchestrator / LLMOrchestrator / UIState) is no longer required by audit findings. It may still be desirable for cognitive-load reasons, but the "god object" urgency is gone.
- **Test coverage is still missing** (0 tests for AppState directly). The layering argument shows the *structure* is not broken; tests are still warranted to *prevent* future breakage.

### 11.2 `ResultMerger` is a broken ranking system (severity: HIGH) — revised after provider-score verification

**Revised 2026-06-17 after re-reading the three provider `score` functions.** The original §11.2 framed this as a "ranking model mismatch" with one +100 boost. The reality is worse: **all three providers assign scores on incompatible scales, and `ResultMerger` sorts them as if they were directly comparable.** The system works only because, in practice, a single query rarely hits multiple providers for the same URL.

- ✅ VERIFIED — `ResultMerger.merge()` (21 lines, `Sources/AISpotlightKit/ResultMerger.swift`):
  ```swift
  for bucket in buckets {
      for r in bucket {
          if let existing = byURL[r.url], existing.score >= r.score { continue }
          byURL[r.url] = r
      }
  }
  return byURL.values.sorted { $0.score > $1.score }
  ```
  Dedup by URL (keep higher score); sort by `score` descending. **No normalization, no per-provider weighting, no semantic awareness of which provider produced the score.**

- ✅ VERIFIED — provider score functions, all read 2026-06-17:

  | Provider | Score formula | Range | Source |
  |---|---|---|---|
  | `FileSystemProvider` | `Double(count - i)` (line 52) | `0 .. N-1` (≈ 0..20 with default limit) | `Sources/AISpotlightKit/Providers/FileSystemProvider.swift` |
  | `ContentSearchProvider` | `Double(count - i) + 100` (line 89) | `100 .. 100+N-1` (≈ 100..120) | `Sources/AISpotlightKit/Providers/ContentSearchProvider.swift` |
  | `AppProvider` (search path) | `entry.r.score + (isPrefix ? 100 : 10)` (line 90); `entry.r.score` is 0 for cached results (line 104) | `10` (substring hit) or `100` (prefix hit) | `Sources/AISpotlightKit/Providers/AppProvider.swift` |

  The `+100` boost in `ContentSearchProvider` is not a one-off anomaly. It appears in **two of three providers** (ContentSearch's base offset, AppProvider's prefix-hit boost). And `FileSystemProvider`'s 0..20 range does not overlap with either.

**Concrete consequences (all derived from the verified ranges):**

- **A `FileSystemProvider` match can never outrank a `ContentSearchProvider` match for any URL** — `FileSystem` max ≈ 20, `ContentSearch` min = 100. Even the *worst* ContentSearch result beats the *best* FileSystem result.
- **A `ContentSearchProvider` content match is indistinguishable from an `AppProvider` prefix match** — both score 100 (plus a small `count - i` adjustment). When a query hits both (e.g. the user's typed name is also a content word), the merger cannot tell which source is more relevant.
- **`FileSystemProvider` results are systematically deprioritized.** Any query where the user wanted a *file* (filename match) but the file also has matching *content* will surface the content match first. The user may not want this.
- **`AppProvider` non-prefix substring hits (score = 10) lose to `FileSystemProvider` top results (score up to 20).** This is the only case where FileSystem can outrank another provider, and it produces a counter-intuitive ranking: an app whose name *contains* the query ranks below a file whose name *equals* the query.

**Why the system appears to work today:** most queries do not produce hits from multiple providers. A filename search typically hits `FileSystemProvider` only; an app name search hits `AppProvider` only; a content search hits `ContentSearchProvider` only. The broken ranking only manifests when providers overlap (rare in the current `QueryParser` output, but possible and unhandled).

**Recommended action (not taken in this audit):**
- (a) **Define a shared `ProviderScore` type** with explicit `(relevance, confidence, source)` fields. Have each provider produce it; have `ResultMerger` sort on `relevance` (or a weighted combination) only. This is the correct long-term answer.
- (b) **Normalize at the boundary**: each provider returns a score in `[0, 1]`, and `ResultMerger` operates on that. `ContentSearchProvider`'s "content > filename" intent becomes a per-bucket weight, not a per-result score offset.
- (c) **Document the broken contract** in `ResultMerger.swift` and accept that ranking is best-effort. Cheapest, but freezes the bug into the codebase.

This finding is the **strongest argument** for not pursuing Step-2 (SQLite augmentation) or Step-3 (merge engine) until the ranking contract is defined. Adding a new provider to a fan-out that is already broken by score-scale mismatch will silently compound the problem.

This was not visible in the original audit (§1–§10) because I had read `ContentSearchProvider` and `ResultMerger` but not `FileSystemProvider` or `AppProvider`. Reading those two made the picture worse, not better.

### 11.3 Implication for the suggested follow-ups (§10)

§10's #1 (fix README drift) and #2 (decide on `LLMIntentRouter` future) are still correct and remain the lowest-cost actions. The two new risks in §11.1 and §11.2 are **higher-priority structural concerns** but also higher-cost to address. They do not block the README fix; they do block the Step-2 SQLite sync if the user (or the next LLM session) intends to wire SQLite into `AppState` directly.
