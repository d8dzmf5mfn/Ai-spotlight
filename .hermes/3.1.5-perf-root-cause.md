## Summary

The initial-content-index walk that hung at 5GB RSS / 100% CPU
on a typical 80k-file `~/Documents` was diagnosed as
**primary-inverted-index size, not actor hop overhead**.

## Live measurements

Added a `Log.write` instrumentation to `IndexStore.upsert` that
counts `documents.count`, `terms.count`, and
`documentToTerms.values.reduce(0) { $0 + $1.count }` (total
postings) every 1000 upserts. Ran the live .app on
`~/Documents + ~/Downloads` (~80k files).

| docs | terms | postings | RSS (app) |
|------|-------|----------|-----------|
| 1,000 | 61,090 | 211,655 | (rising) |
| 10,000 | 190,312 | 1,778,462 | (rising) |
| 70,000 | 383,204 | 12,767,921 | (rising) |
| 74,000 | 460,668 | 13,602,748 | 4.9 GB |

The 74k-docs case:
- **460k unique terms** (~6.2 terms per doc on average across
  the corpus; skewed by code files which have 50+ tokens each)
- **13.6M postings** in the inverted index
- Each posting is a `URL` in `Foundation`, which carries:
  - NSURL bridge
  - CFURL
  - path components array
  - cached absolute / relative paths
  - ARC overhead
  - at least ~80â€“200 bytes per URL, often more
- 13.6M Ã— 100 bytes â‰ˆ **1.3 GB just for the URL values**, before
  dict / set bucket overhead. With ARC and dict metadata this
  doubles to **2â€“3 GB**, matching the observed 4â€“5 GB RSS.

The bulk of the memory is in `terms: [String: Set<URL>]`. The
per-call actor hop is real but contributes a small fraction
(< 200 MB) by comparison.

## The four candidate causes (ranked)

| # | Hypothesis | Probability | Verdict |
|---|------------|-------------|---------|
| 1 | Inverted index `[String: Set<URL>]` is large (8M+ postings) | 70% | **Confirmed by measurement** |
| 2 | `TextExtractor.tokenize` allocates big intermediate arrays | 20% | **Ruled out**: single-pass O(n) with `currentTerm` accumulator; `out` array is proportional to input size |
| 3 | 80k `await store.upsert` causes repeated Dict/Set rehashing | 8% | Real (small constant factor); fixed in part by `ingestInBatches` (commit 15c609c) |
| 4 | Actor hop copy overhead | 2% | Real but bounded; not the cause of the 5GB |

The instrumentation code is not in the repo (commit `e5014f6`
removed it). This file is the only record of the measurement.

## The fix (Phase 4)

Two structural changes:

### 1. `bulkUpsert` for batched actor hops (perf, not RSS)

```swift
public struct PendingDocument: Sendable {
    let doc: IndexDocument
    let terms: Set<String>
}

public func bulkUpsert(_ batch: [PendingDocument]) {
    for item in batch {
        // ... same body as upsert ...
    }
    built = Date()
}
```

`ContentIndexer.ingestInBatches` then calls
`await store.bulkUpsert(chunk)` once per 1000-file batch,
reducing 80k actor hops to 80. The current TaskGroup pattern
(`ingestInBatches` + per-file `await ingest`) still does 80k
hops under the hood â€” this fix is necessary but not sufficient.

### 2. Replace `URL` with `DocID: Int` (the real RSS fix)

```swift
public typealias DocID = Int

// Replace the doc table:
private var documents: [DocID: IndexDocument] = [:]
private var documentToTerms: [DocID: Set<String>] = [:]
// Replace the inverted index:
private var terms: [String: Set<DocID>] = [:]
// Persist a URLâ†’DocID map for the rare case where we need
// the URL given a DocID (i.e. when returning search results).
private var urlToDocID: [URL: DocID] = [:]
private var docIDToURL: [DocID: URL] = [:]
```

Size comparison for 80k docs:
- 13.6M postings Ã— 4 bytes (Int) â‰ˆ 55 MB for the posting lists
- 80k docs Ã— ~200 bytes (URL) â‰ˆ 16 MB for the doc table
- **Total ~70 MB**, down from 4â€“5 GB.

This is the standard search-engine layout. The trade-off is
that query results return `DocID` which must be resolved to
`URL` via `docIDToURL[docID]`. A `Map` lookup is O(1), so the
hot path is unchanged.

## Test impact

The query-side `IndexStore.query(...)` and `IndexHit` API
both already use `URL` as their public type. To keep the
DocID refactor internal:
- Keep `IndexHit.url` as `URL`
- Internally: `query` returns `DocID`s, then `IndexStore` maps
  back to `URL` via `docIDToURL` before exposing

Existing 9 `IndexStoreTests` and 10 `ContentSearchProviderTests`
should pass without modification if this is done cleanly.

## Why not fix it now

- Token budget is at ~7%
- DocID refactor touches 4 files (`IndexStore`, `IndexSnapshot`,
  `ContentIndexer`, `ContentSearchProvider`) and 5 tests
- bulkUpsert is straightforward but needs the same code review
- Better to ship Phase 3.1 with the symptom clearly documented
  (this file) and the fix roadmap in Phase 4
