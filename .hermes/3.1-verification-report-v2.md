### (g) Ingest phase completes — ⛔ **Fails — 5GB RSS, never finishes**

After walk completes, the ingest phase starts. It calls
`IndexStore.upsert` for each of the 83,590 files. We tried
three implementations:

1. **Sequential** (`for d in toUpsert { await ingest(d.url) }`):
   Never completed in 3 minutes. CPU 100%, RSS 5GB, growing.
2. **TaskGroup batch=500**: Never completed in 60 seconds. CPU
   140% (parallel), RSS 5GB.
3. **TaskGroup batch=50**: Same. CPU 100%, RSS 5GB.

The RSS is identical across batch sizes, which means the
memory blow-up is **not** caused by parallelism per se. It
looks like a leak or massive per-file allocation in
`IndexStore.upsert` or in `String(data: contentsOf)` + tokenize.

The walk decides "ingest" for ALL 83,590 files (none had a
matching mtime in the cache because the index file didn't
exist). At ~1GB of RSS per ~5–10k files, the linear growth
doesn't match a true leak. It's more likely a per-file
allocation that the runtime can't free fast enough.

## What's Known Broken / Incomplete

**Ingest is unusable at scale.** The fix is structural, not a
batch-size knob:

1. **Avoid reading the file at all in Phase 3.1.** Index the
   *path* (and basic mtime + size) and tokenize lazily on
   query. The current "index then query" model assumes the
   indexer is fast and the queryer is the expensive part,
   but for 80k files the opposite is true.

2. **Batch the upsert.** If we keep reading, the upsert
   itself should take a `[IndexDocument]` array and write
   them all in one actor call, not one per file.

3. **Use a real inverted-index builder (SQLite or LMDB)** for
   persistence. JSON is fine for the prototype but doesn't
   scale.

This is a Phase 3 polish / Phase 4 architectural fix. For now,
Phase 3.1 is "wired up" but the user-facing search is "empty
until first indexing completes, which on a typical Documents
dir may take 5+ minutes and use 5GB RSS."
