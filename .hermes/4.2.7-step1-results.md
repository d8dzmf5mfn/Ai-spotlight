## Live Test Results

### Test environment
- 16 docs (Documents only, pre-Phase-4.2.7): RSS 133MB,
  index.json 152KB
- 83608 docs (Documents + Downloads + Desktop):
  - RSS: **5.3GB**
  - index.json: **204MB**
  - walk: 15.73s
  - ingest: 90s

### What worked
- **On-disk index 14x smaller** (2.8GB → 204MB). The JSON
  format is more compact: Int32 keys + Int32 postings
  vs URL keys + URL postings.
- **Walk + ingest + persist all completed** for 80k files
  in ~2 minutes — the previous Set<URL> version
  effectively never finished persisting.
- **All 117 tests pass** with the new schema.

### What didn't work
- **RSS didn't shrink from 5GB → 500MB as the plan
  predicted.** RSS stayed at 5.3GB. Reason: Swift's
  `Set<Int32>` is a hash table with 20-40 bytes per entry
  (bucket overhead, load factor, value type boxing), not
  the 4 bytes per entry the plan assumed.
- The plan's math was based on a 4-byte Int32 per posting
  with 25M postings = 100MB. The real number is 25M × 30
  bytes (real Set hash table entry size) = 750MB for the
  Set itself, plus the Set's load factor overhead, plus
  dict wrapping (the [String: Set<Int32>] has its own
  hash table around the sets) = 5GB.

## What's needed to actually hit 500MB

The plan was right that 4-byte postings would fit, but
Swift's standard collections add hash table overhead that
swamps the per-element size. To actually hit 500MB we
need to skip the `Set` and use a more compact structure:

### Option 1: Sorted [Int32] per term, binary search for query
- 4 bytes per Int32, no hash table, no bucket overhead
- Insert: O(N) (insert at sorted position) per term,
  amortized O(N²) for bulk
- Query: O(log N) per term, O(K log N) for K query terms
- For our use case (top-5 results, not all matches) this
  is fast enough
- Expected: 25M postings × 4 bytes = 100MB + dict
  overhead = 200-300MB. **Big win, modest refactor.**

### Option 2: Bitmap per term
- 1 bit per document, so for 80k docs it's 80k bits =
  10KB per term. Very memory-efficient.
- Query: O(K) bitwise AND/OR across K terms, then iterate
  set bits
- Doesn't scale to 500k docs gracefully (62.5KB per
  term, 460k terms = 28GB — not viable)

### Option 3: Compressed inverted index (Lucene-style)
- Variable-byte or PFOR-Delta encoding
- 1-2 bytes per posting on average
- 25M postings × 1.5 bytes = 38MB, plus overhead = 100-200MB
- Most aggressive savings, but most code

### Option 4: Just go to SQLite FTS5 (the originally
recommended Step 2-3)
- Skip the in-memory optimization
- Let SQLite handle the mmap + page cache + compression
- Native to macOS (no SPM dep)
- 500k docs = ~50MB RSS (SQLite's working set)
- This is the destination anyway

## Recommendation

Given that:
1. The schema v2 (DocID keying) is correct and shipped
2. The 14x JSON size win is a real win
3. The RSS win didn't materialize due to Swift's Set
   hash table overhead
4. The path from here to 500MB is one of:
   - sorted-array refactor (1-2 hours, hits ~300MB)
   - SQLite FTS5 (the real destination, 4-6 hours,
     hits ~50-100MB)
5. The user wants full-disk indexing (200k-500k files)

**The honest recommendation is: skip the sorted-array
refactor, go straight to Step 2-3 (SearchBackend protocol
+ SQLite FTS5 backend).** The DocID schema is the right
shape; the in-memory storage is just the wrong place.
SQLite gives us:

- Disk-backed (no RSS bound by RAM)
- mmap (page cache does the right thing automatically)
- FTS5 (tokenized, scored, ranked — all the things we'd
  have to hand-roll)
- Compressed inverted index built in (typically 10-100x
  smaller than equivalent in-memory structures)
- Mature crash recovery (WAL)
- Already on every Mac (no SPM dep)

The DocID schema is forward-compatible: the SQLite
backend will use the same DocID for its own doc table,
just like IndexStore does. The public API
(`upsert(_:terms:)`, `query(_:limit:)`) is preserved. We
just swap the backend implementation.
