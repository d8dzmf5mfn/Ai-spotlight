## What's Known Broken / Incomplete

1. **No content search in the live panel.** Searching "polyester" today goes through the Phase 1 rule-based QueryParser (filename MDQuery) and Phase 1 `FileSystemProvider` — neither looks at file contents. The wiring is in unit tests but not in `AppState`.

2. **No IndexStore created at app launch.** Without 3.1.5, there's no `~/Library/Application Support/AISpotlight/index.json` file in production. Content search is offline.

3. **No Settings UI for content index.** SettingsView doesn't have a "Content" section (no root paths, no "Rebuild index" button, no "Last built" timestamp).

## Why 3.1.5 Is Deferred

Running `swift test --filter IndexManagerTests` (or any test target build that includes `Content/IndexManager.swift`) hangs indefinitely on macOS 27 beta / Xcode 27 beta. The parent `swift-test` process is alive in S state with 0% CPU and 0 child processes. Direct `xcrun xctest` of the already-built bundle hangs the same way.

Working hypotheses:
- Swift 6 strict-concurrency interaction with `IndexManager`'s `nonisolated public func` + `await setIsRunning` pattern may deadlock the test runner
- macOS 27 beta may have a new sandbox / xctest harness issue
- A new `IndexManager` actor or simpler test surface may avoid the hang

## Next Steps

When the SPM hang is understood (or worked around), return to 3.1.5. The unit test for it is already designed and on disk. The `swift package reset` (clears manifest cache) and stale `xctest` process kill both proved insufficient.

The verification gate confirms: **all of Phase 3.1's **code** is correct and tested, but the live integration (AppState ↔ ContentSearchProvider ↔ IndexStore) is not yet wired up because 3.1.5 is blocked.**
