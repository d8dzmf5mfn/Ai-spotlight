## Core Phase 4 Skills (created today)

### swift-actor-bulk-call-pattern
Created during Phase 4.x. The pattern of batching many
small actor method calls into one bulk method. Saves
80k actor hops → 80 hops on indexing 80k files.

### parse-json-from-small-models
Robustly parse JSON from local small LLMs (gemma2:2b,
phi3, etc.) that wrap in prose, fence in markdown, omit
fields, or send wrong types. 4 failure modes documented,
30-line parser, 6 unit tests.

### llm-intent-router-pattern
The Siri/Copilot/Notion AI approach: ask a small LLM to
classify user intent into structured JSON, fall back to
rules. Includes three-tier routing (cache → LLM → rules)
and the known gotcha that the LLM router hammers Ollama
with one call per keystroke.

### production-runtime-instrumentation
Add boundary logs at the right places (% 1000 == 0
pattern, or per-call). When something's slow or stuck,
these logs are how you figure out which step is the
culprit without spelunking through the call stack.

### inverted-index-url-vs-int-pattern
The classic trap: storing URL objects in an inverted
index (Set<URL> per term) instead of Int DocIDs. URLs are
80-200 bytes each in Foundation. With 80k files, 13M
postings × 100 bytes = 1.3 GB just for the URL values. The
fix is the standard search-engine layout: DocID as Int,
URLs only in the doc table. 30x reduction. Phase 3.1.5 of
AI Spotlight is stuck on this.

### enter-as-commit-vs-typing-as-preview
In any UI with both cheap preview (typing) and expensive
action (LLM call, network, computation), typing should
NEVER auto-fire the slow path. Enter is the commit. The
AI Spotlight Phase 4.2.x fix: removed auto-ask on debounce,
gated it on Enter.

### recursive-walk-symlink-cycle-detection
When writing a recursive filesystem walker, add
`visitedDirectories: Set<URL>` cycle detection. macOS has
symlinks like `~/Library` → `/private/var/...` and the
`com.apple.metadata.index` Spotlight indices that look
like infinite loops if you don't track them. One line of
code: `if visitedDirectories.contains(dir.standardized) {
return }`.

### text-content-indexing-patterns
The full reference for building a local content index:
tokenize, index file extensions, persist as JSON, what
goes in the inverted index vs the doc table, the
default-allow-list of extensions (md, txt, swift, py, etc).

### tdd-stub-when-third-party-unavailable
How to unit-test code that depends on an AppKit-only
class: define a one-method protocol, write a fake
implementation in tests. The `OrchestratorContentTests`
file uses this pattern to test the orchestrator without
spinning up an NSTextField.

### integration-regression-testing
The three categories of integration regression:
1. **Instantiated but not wired** — @Published field
   set but the view doesn't bind to it
2. **Try? silent fallthrough** — `try?` swallows the
   error, the path silently takes the wrong branch
3. **Debounce/cancel race** — debouncing 0.6s of typing
   races against the slow path

Plus the diagnostic log pattern: one log at the
boundary, one log inside, so you can see which path
was taken.

### file-logger-fsync-for-debug
When writing a debug logger to a file in /tmp, add
`try? h.synchronize()` between write and close. Otherwise
the OS buffers and your debug logs are invisible during a
live reproduction — they only flush on quit. A single
line; costs nothing measurable.

### log-cancellation-source-vs-failure-type
When debugging an async system, the error code tells you
who cancelled the work. NSURLError -1004 = host
unreachable (network). NSURLError -999 = cancelled
(someone called Task.cancel()). NSURLError -1005 =
network connection lost (Ollama crashed mid-response,
often jetsam-killed). Reading the code first narrows the
search to "whoever called Task.cancel()" instead of "is
the AsyncThrowingStream broken?". This 5-minute read
saved us from going down the wrong rabbit hole for
hours.

### nstextfield-controltextdidchange-fire-timing
NSTextField fires `controlTextDidChange` AND
`control(_:textView:doCommandBy:)` on the same runloop
tick when the user presses Enter. controlTextDidChange
fires FIRST with the final text value, then Enter. Plus
NSTextField can fire controlTextDidChange a second time
on the same tick when the panel is being cleared (the
panel-clear path). This is invisible unless you log it.
The fix: do NOT cancel expensive in-flight work in
onChange — NSTextField fires it on the same tick as
Enter, and you'll race against the Enter-spawned
expensive work.

### module-split-appkit-test-deadlock
Swift Concurrency test target + `import AppKit` = test
target hangs in Phase 1.1.0 of AI Spotlight for 6 minutes
before we realized the cause. The fix: split Kit into Core
(Foundation only) + Mac (AppKit). Tests run against Core.
Real AppKit-requiring code goes in a separate MacTests
target. The same trick fixes `xctest` hanging on
macOS 27 beta.

### macos-swiftpm-bug-hang
When `swift test` (or `xcrun xctest`) hangs indefinitely on
macOS 27 beta / Xcode 27 beta, the test target is dead in
the water. Symptoms: parent alive in S state, 0% CPU, 0
child processes, no stack trace. Workarounds: kill any
zombie xctest/swift-test processes, try
`swift package reset` (preserves .build/), or as a last
resort `rm -rf .build && swift test` (but watch out for
the GitHub network fetch on the rebuild).

### pdf-text-extraction
Extracting text from PDF files via PDFKit. The trap:
`PDFDocument.string` does NOT read PDFAnnotation contents —
only the underlying text stream. To get text in
test-generated PDFs, you have to use Core Graphics +
CTLineDraw, or use a pre-existing real PDF as a fixture.

### notification-observer-lifetime
`_ = MyObserver()` releases the observer immediately. The
NotificationCenter callback never fires. Always hold the
observer in a property of the owning class. Phase 1.2 of
AI Spotlight hit this in a Settings window.

### swift-release-logging
Debug logging that actually works in release builds
(NSLog gets stripped). The pattern: `Log.write(_:)` to a
file in /tmp via per-call FileHandle. See
`file-logger-fsync-for-debug` for the fsync enhancement.

### search-as-command
When you can't add a Settings menu (`.accessory` app),
hide commands in the search bar. AI Spotlight has
`CommandMatcher.match("settings")` that returns a
`SearchResult.command(command: .openSettings)`. Activate
posts a NotificationCenter event the settings window
listens for. The user can search "settings", "quit", etc.
and the result row, when activated, runs the command.

### code-review-audit-checklist
Systematic dead code + perf audit. The 6 categories
(dangling code, dead methods, unused imports, duplicate
logic, perf hot spots, memory leaks). When you've been
moving fast, this is the hourly-rate job that catches
the things you wouldn't notice by re-reading.

### switpm-app-bundle-packaging
Why SwiftPM's `.executableTarget` doesn't produce a .app
bundle, and the manual script (Info.plist + ad-hoc
signing + PkgInfo) needed to make one. Includes the
`Bundle.module` workaround for SwiftPM executable targets
that need resource bundles.

### search-provider-protocol-extension
The SearchProvider protocol that lets the orchestrator
fan out to file / app / content providers. Adding a new
provider type (PDF, calendar, contacts, …) is a 1-line
change in main.swift — no orchestrator changes needed.

### swift-app-entry-point
The `main.swift` pattern for SwiftPM .executableTarget.
Why `@main` doesn't work, why you need the
`final class AppLauncher: NSObject, NSApplicationDelegate`
bootstrap, and why `NSApplication.shared.run()` is a
forever call (you put all your setup before it).

### swift-inverted-index-perf-design
The architectural reference for designing an inverted
index. URL vs Int DocID, per-doc term map vs posting list,
JSON vs SQLite for persistence, when to bulk-insert, when
to lazy-load. The doc table vs posting list is the central
distinction.

### fileurl-symlink-comparison
When comparing two URLs that should refer to the same
file but `==` returns false (even when the path strings
look identical), use `url.standardizedFileURL` to
canonicalize before comparing. macOS has symlinks
(`/var` vs `/private/var`, `/tmp` vs `/private/tmp`)
that make naive comparison fail.

### phase-completion-verification-report
After any "wired up" commit, the verification gate is:
1. `swift build -c release` — clean
2. `./scripts/make_app.sh` — assemble
3. `open build/YourApp.app` — start
4. Actually use the app for 60 seconds — click the
   buttons, type the queries, watch the panel
5. `tail -f /tmp/yourapp.log` while you do
6. If anything in the user path fails, find it in the
   log first, before changing code

This is not optional. "Wired up" without "exercised live"
is a half-claim. Most "wired up" commits need a fix
within 1-3 days of shipping.
