## Bug 1: "After 2 conversations, can't continue dialog"

**Symptom (user-reported):**
> 进行了两次对话后无法继续对话不知道为什么

**Log evidence (real reproduction, today):**
```
14:02:02 [AppState] runLLMAsk: stream done, llmReply length=3480   ← conversation 1
14:02:04 activate: ask intent, q="...is a file", contextURLs=0
14:03:21 [AppState] runLLMAsk: stream done, llmReply length=453    ← conversation 2 (shorter!)
14:03:31 ...find the specific location of this folder in the finder...
14:03:39 activate: no selection and no ask intent, doing nothing   ← BROKEN
14:03:45 activate: no selection and no ask intent, doing nothing   ← BROKEN
14:03:55 activate: no selection and no ask intent, doing nothing
14:03:58 activate: no selection and no ask intent, doing nothing
```

**Suspected root cause:**
The third turn starts with "find the specific location of
this folder in the finder". `QueryParser` matches the
"find" verb → returns `Intent.findFile`. `runSearch` fans
out to `FileSystemProvider.search(intent:)` which returns
empty (no file matches the name). `activate()` priority 1
sees no selection (no result for user to pick), priority
2 only has fall-back for `.ask` and `.openApp` — `.findFile`
silently falls through to priority 3 which does nothing.

The user *intended* this to be a conversation follow-up
("find that thing I just told you about"). The LLM could
have answered it with the conversation history. But the
rule parser doesn't have the context.

**Intended fix (not yet committed — see also Bug 2):**
Add a third arm to `activate()` switch on `Intent.findFile`:
if `results.isEmpty`, fall back to `runLLMAsk(query, contextURLs: lastSearchContextURLs)` so the LLM can interpret the question with prior context. The diff was written but the build process was killed before commit:

```swift
case .findFile:
    if results.isEmpty {
        Log.write("[AppState] activate: findFile with no match, falling back to LLM ask")
        await runLLMAsk(query: query, contextURLs: lastSearchContextURLs)
        self.query = ""
        return
    }
```

**Test plan:** Type conversation 1 ("tell me about X"), wait for reply, type conversation 2 ("tell me more about X"), then conversation 3 ("find that file in finder") — verify the third turn fires the LLM with a non-empty reply instead of silently no-op'ing.

**Phase:** 4.2.5 (immediate).

## Bug 2: "AI can't search my Finder"

**Symptom (user-reported):**
> ai 好像无法搜索我的 finder(好像不支持)

**Status:** **NOT a bug. By design.**

**Explanation:**
The user's mental model is: "AI Spotlight = Spotlight with
AI = AI can do whatever Spotlight can, plus thinking." In
fact, AI Spotlight's architecture is:
- `FileSystemProvider` does the Finder search (MDQuery, same
  engine Spotlight uses)
- `ContentSearchProvider` does in-memory content search
  (currently broken — see Bug 3)
- `AppProvider` does the app launcher
- `LLMConversationService` is a separate path that the LLM
  flows through

The LLM is **not** connected to `FileSystemProvider` or
`ContentSearchProvider`. When the user types "find X in
finder" and presses Enter, the LLM gets a `query: "find X
in finder"` prompt and either:
- Returns text saying "I cannot access your filesystem"
- Or hallucinates a path

The user wants: "I tell the LLM a question, LLM uses tool
calling to query the file system, returns results." This
is the **tool-use / function-calling** pattern that GPT-4
and Claude use. It requires:
1. Defining tools (e.g. `search_files(query: String) ->
   [FileMatch]`) in the system prompt
2. Parsing tool-call requests from the LLM response
3. Executing them against the SearchOrchestrator
4. Sending results back for a second LLM turn
5. Final LLM turn with the tool results

This is a substantial architectural change. Estimated
3-5 hours of work for Phase 4.3.

**Intended fix (Phase 4.3):**
Add `LLMTool` protocol with implementations
`SearchFilesTool(query:limit:) -> [FileMatch]` and
`OpenAppTool(name:) -> URL?`. Add tool definitions to the
LLM system prompt. In `LLMConversationService.ask`, parse
the LLM response for `tool_use` blocks, execute them,
loop back to the LLM with the tool results, repeat until
the LLM emits a final text response. Cap at 3 tool
iterations to prevent infinite loops.

**Test plan:** Type "find my polyester notes" — the LLM
should emit a `tool_use: SearchFilesTool(query: "polyester")`,
the system should execute the search and return file
matches, the LLM should produce a final answer naming
specific files with paths the user can click.

## Bug 3: Content search returns empty (Phase 3.1.5 perf bug)

**Symptom (corollary of Bug 1):**
User asked the LLM to find files about polyester; the LLM
returned 3480 chars of text but no specific file paths.
When asked to "find the location" of "the AI Spotlight I
mentioned", the rule parser classified it as `.findFile`
but the file system search returned empty.

**Root cause (KNOWN — documented in `.hermes/3.1.5-perf-root-cause.md`):**
The in-memory inverted index (`IndexStore`) uses
`[String: Set<URL>]` for the term→postings map. With 80k
files indexed, this consumes 4-5 GB RSS and the
`persist()` writes 2.8 GB to disk. We disabled
`~/Downloads` from `IndexManager.defaultRoots` to keep the
memory bounded at 130 MB, but with only `~/Documents` in
the index, the inverted index is too small to be useful.

Content search via `ContentSearchProvider` returns 0
matches for almost any query that isn't an exact filename.

**Intended fix (Phase 4.x):**
1. **DocID refactor** — replace `String -> Set<URL>` with
   `String -> Set<DocID: Int>`, where `DocID` is the
   numeric key in a separate `[Int: IndexDocument]` table.
   The URL stays only in the doc table, not in the inverted
   index. Expected: 1.3 GB → 55 MB for the posting lists.
2. **Bulk upsert** — replace per-file `await upsert(...)`
   with batched `bulkUpsert([DocID, terms])` to amortize
   actor hop overhead.
3. Re-enable `~/Downloads` after the memory blow-up is
   fixed.

Estimated: 1-2 hours of work, mostly in
`Sources/AISpotlightKit/Content/IndexStore.swift` and
`ContentIndexer.swift`.

## Bug 4: "AI can't remember what I said earlier"

**Symptom (corollary of Bug 1 + Bug 3):**
The LLM has no memory of prior conversation turns. The user
types "tell me about polyester" → LLM answers. Then
"find the location of it" → LLM doesn't know what "it"
refers to.

**Root cause:**
`LLMConversationService.askWithHistory` exists and works
in unit tests, but `AppState.runLLMAsk` doesn't pass
`history:` to it. The prior turn's reply lives in
`llmReply` (the @Published field), but the field is
overwritten on the next turn — there's no `history: []`
array being built up.

**Intended fix (Phase 4.2.6):**
- Add `@Published var llmHistory: [HistoryEntry] = []` to
  AppState
- Before `runLLMAsk`, append the current query as a user
  HistoryEntry and the previous `llmReply` (if any) as an
  assistant HistoryEntry
- Pass `history: llmHistory` to `runLLMAsk`
- After the ask completes, append the new reply to
  `llmHistory` (capped at 6 turns = 12 messages to stay
  under small-model context windows)

Estimated: 30 minutes of work.

## Bug 5: "Ollama 突然自己 quit" (memory jetsam)

**Symptom (user-reported):**
> 为什么ollama会突然自己quit呢

**Status:** **Root cause identified, partially mitigated.**

**Root cause:**
macOS `duetactivityscheduler` runs memory-pressure scoring
every ~5 seconds. When Ollama is loading or running a 12B
model (8 GB model + 1-2 GB KV cache = ~10 GB resident), and
the system is also running AI Spotlight + the user's other
apps, the combined memory pressure can cross a threshold.
macOS jetsam then SIGKILLs the largest consumer — Ollama.

**Evidence (from `log show`):**
```
21:55:18 ... Ignoring jetsam update because this process is not memory-managed
21:55:19 ... com.apple.duetactivityscheduler:scoring ... memory-maintenance.report-jetsam-telemetry
21:55:19 ... [app<application.com.aispotlight.app>:7435] Ignoring jetsam update ...
21:55:25 ... [app<application.com.aispotlight.app>:7435] Ignoring jetsam update ...
21:55:30 ... [app<application.com.aispotlight.app>:7435] Ignoring jetsam update ...
```

**Fixes applied:**
- `friendlyLLMError` now distinguishes -1005 "Ollama
  crashed mid-response" from -1004 "Ollama is not
  running" so the user knows to check `ollama ps`.
  Committed in `60256ed`.
- `LLMIntentRouter` disabled by default (was hammering
  Ollama with one call per keystroke). Committed in
  `1181c74`.

**Fixes still needed (Phase 4.2.6):**
- Settings UI to let user pick a smaller model
  (`gemma2:2b` is 1.6 GB, jetsam-safe; `gemma4:12b` is
  8 GB, will jetsam under memory pressure).
- Recommended: `ollama pull gemma2:2b` for now.
- Optionally: a hard-coded `OLLAMA_KEEP_ALIVE=24h` so
  Ollama doesn't unload the model between requests.
